# cc-switch CLI UI Refresh Design

## 背景

`cc-switch` 当前已经具备完整的 profile 管理能力，但终端界面的信息层次偏弱：

- 普通输出基本是线性文本，`list/current/status` 的信息密度和可扫读性不一致。
- 交互式列表主要依赖 `> ` 前缀区分选中项，当前配置、可执行动作和危险操作的提示不够明显。
- `list` 与 `status` 的交互界面视觉语言不统一，用户需要重新理解页面结构。

本次优化只针对 CLI 展示层和交互层，不改变 profile 持久化、settings 写回、错误处理语义和命令集合。

## 目标

1. 提升普通输出与交互界面的扫读性，让用户更快识别当前配置和可选动作。
2. 在支持 ANSI 的 TTY 中加入克制的颜色和分组标题；非 TTY 输出保持纯文本稳定。
3. 保留当前两步式交互模型，同时为熟练用户补充快捷键。
4. 统一 `list` 与 `status` 的终端视觉语言，但避免引入高风险的大改版。

## 非目标

- 不新增命令，不重做交互框架，不引入第三方 TUI 依赖。
- 不修改 `use/add/edit/remove/rename` 的业务流程。
- 不改变非 Darwin 平台的交互能力边界；不支持 raw terminal 的平台继续使用现有纯文本路径。
- 不引入复杂 box-drawing 或高度依赖终端宽度的布局。

## 约束

- 只在 `stdout` 为 TTY 时启用 ANSI 美化；重定向和管道输出保持纯文本。
- 视觉风格选择“极简专业”：少量颜色、明确分组、克制提示。
- 交互方式选择折中模型：保留 `Enter` 菜单，同时支持 `e/r/d` 快捷键。
- 实施时每个小任务修改文件数不超过 3 个。

## 方案比较

### 方案 A：保守增强型

- 在现有渲染和输入模型上做增强。
- 统一标题、分组、状态色、选中态和底部提示。
- 对现有测试和行为影响最小。

这是本次选定方案。

### 方案 B：面板型

- 通过更多边框、块状区域和 dashboard 布局强化视觉。
- 可读性更强，但宽度适配、兼容性和实现复杂度明显上升。

### 方案 C：操作优先型

- 重点强化快捷键和即时操作反馈。
- 切换效率更高，但普通输出和交互界面的视觉统一性较差。

## 选定设计

### 1. 视觉语言

- 普通输出采用“两层信息结构”：标题或主状态在上，细节字段和列表在下。
- 颜色仅表达语义：
  - 当前配置或选中项使用主高亮色。
  - 次要说明使用弱化色。
  - 删除确认使用警示色。
- 不使用大面积彩色边框，也不依赖特殊 Unicode 字符。

### 2. 普通输出

涉及 [`internal/output/print.go`](/Users/liuzhiqiang/DevOps/cc-switch/internal/output/print.go) 和新增样式辅助文件。

- `current` / `status` 统一为同一套详情样式：当前配置、接口地址、模型分组显示。
- `list` 从单列裸输出改成分组输出：
  - 当前配置
  - 其他可用配置
- TTY 模式下当前配置高亮，说明文字弱化。
- 非 TTY 模式下仍输出稳定纯文本，不加入 ANSI 转义字符。

### 3. `list` 交互界面

涉及 [`internal/cli/list_menu.go`](/Users/liuzhiqiang/DevOps/cc-switch/internal/cli/list_menu.go) 与 [`internal/cli/app.go`](/Users/liuzhiqiang/DevOps/cc-switch/internal/cli/app.go)。

- 页面分成三段：
  - 顶部：标题和上下文说明
  - 中部：profile 列表或动作列表
  - 底部：快捷键提示
- 选中态不再只靠 `> `，而是组合使用高亮行、当前配置标识和更清晰的文案。
- 保留当前交互路径：`Enter` 打开动作菜单。
- 增加直接快捷键：
  - `e` 编辑
  - `r` 重命名
  - `d` 删除
- 当前配置不显示删除动作，危险操作仍需进入确认页。
- 删除确认页需要突出目标配置名和风险提示，并清晰区分“确认删除”和“取消”。

### 4. `status` 交互界面

涉及 [`internal/cli/status_selector.go`](/Users/liuzhiqiang/DevOps/cc-switch/internal/cli/status_selector.go)。

- 采用和 `list` 一致的标题、字段、底部提示风格。
- 保持更轻的操作模型：上下选择，`Enter` 直接切换，`q` 退出。
- 当前配置始终排在第一位，视觉上明确标注。

## 数据流与行为边界

- 本次改动只影响渲染结果和按键分发。
- 任何切换、重命名、删除、编辑操作，仍然调用现有业务函数：
  - `switchProfile`
  - `runEditWithPromptReader`
  - `runRename`
  - `runRemove`
- 如果操作失败，仍然沿用当前错误输出；本次只确保交互状态恢复和界面提示更清晰。

## 错误处理

- 当终端不支持交互或无法进入 raw mode 时，继续回退到现有纯文本输出。
- ANSI 样式检测失败时默认降级为无样式。
- 空列表、缺失当前配置、空 description、空 model 必须都有稳定退化显示。

## 测试策略

需要新增或更新以下测试：

1. 普通输出在 TTY 与非 TTY 场景下的渲染差异。
2. `list` 交互页的分组标题、当前配置标识和底部快捷键提示。
3. `list` 快捷键 `e/r/d` 的动作分发。
4. 删除确认页的警示文案和取消路径。
5. `status` 页面在空 model、空列表、当前配置优先排序下的渲染。

## 实施拆分

为满足“单次任务不超过 3 个文件”的约束，本次实现拆为三个独立任务：

1. 普通输出样式基础层
2. `list` 交互界面增强
3. `status` 交互界面增强

每个任务都采用先补测试、再最小实现、再验证的方式推进。
